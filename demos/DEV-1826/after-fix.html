Correct ESM/CJS dual-mode type declarations: nested exports, .d.mts wrappers, "type":"commonjs"

← Before Fix After Fix (current) →

Changes Applied

📄
Fix 1 — "type": "commonjs" in published package Added "type": "commonjs" to the generated tmp/package.json (not the source). This makes TypeScript unambiguously resolve .d.ts files as CJS declarations under moduleResolution: "node16". The source package.json was left unchanged to avoid breaking ESM import syntax in test files.
🔄
Fix 2 — Thin .d.mts wrapper files prepare-package-for-publish.mjs now generates a .d.mts wrapper for every .d.ts file. Each wrapper re-exports from its .js sibling, which TypeScript maps to the .d.ts at lookup time. This avoids InternalResolutionError that would occur if .d.ts were copied verbatim (extensionless imports fail under ESM moduleResolution).
🗺️
Fix 3 — Nested exports conditions All typed exports condition objects split from flat { types, import, require } to nested { import: { types: ".d.mts", default: ".mjs" }, require: { types: ".d.ts", default: ".js" } }. ESM consumers now get .d.mts, CJS consumers get .d.ts.
📁
Fix 4 — nodir: true in glob Added nodir: true to glob.sync() so wildcard rules no longer emit directory paths into the exports map. Eliminated 5 publint errors.
🔒
Fix 5 — Stricter export validation Upgraded from existsSync() to statSync(...).isFile() so directory targets are correctly rejected at publish-prep time, not silently allowed through.
🛡️
Fix 6 — CI gates added verify-emitted-types.yml now runs publint@0.3.21 and @arethetypeswrong/cli@0.18.3 on every PR against the built tmp/ directory. FalseExportDefault is explicitly ignored (Babel CJS interop artefact).
