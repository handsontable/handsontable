PR #12482 - Numeric cell European locale dot-thousands parsing

Build: Released v17.0.1 BUG PR #12482 Fix FIXED
BUG present (v17.0.1). Numeric cells using a European locale (locale: 'de-DE', decimal separator ,, thousands separator .) mis-parse dot-grouped values:
  • 7.000 → stored as 7 (should be 7000)
  • 1.234.567 → stored as 1 (should be 1234567)
  • 7.000,25 → stored as a string (should be 7000.25)
Root cause: getParsedNumber had no path for dot-thousands grouping; dots were treated as decimal points, truncating everything after the first .. 
// BUG — helpers/number.js getParsedNumber (before fix):
// No path for dot-thousands. "7.000" with comma decimal hits the float path:
const unifiedNumericData = normalizedNumericData.replace(',', '.');
// "7.000" → replace nothing → parseFloat("7.000") → 7  (dot treated as decimal!)
// "1.234.567" → parseFloat("1.234.567") → 1  (stops at second dot)
// "7.000,25"  → replace comma → parseFloat("7.000.25") → 7  (two dots)
How to observe the bug:
  1. Click "Run test cases" to programmatically set European-style values.
  2. BUG: getSourceDataAtCell shows 7 instead of 7000, 1 instead of 1234567, etc.
  3. You can also double-click any cell in the grid and type a value like 7.000 then press Enter.
  4. Compare with the Fix page.
