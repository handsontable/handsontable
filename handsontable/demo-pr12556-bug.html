PR #12556 — Autofill silently blocked with undefined properties

Build: Released v17.0.1 BUG PR #12556 Fix FIXED
BUG present (v17.0.1). Autofill silently stops working after the first fill when cell data is an object with undefined-valued properties (e.g. { id: 1, label: undefined }).
  • 1st autofill (row 0 → row 1): succeeds. ✓
  • 2nd autofill (row 1 → row 2): silently does nothing. ✗
Root cause: deepClone uses JSON.parse(JSON.stringify(obj)) which drops undefined-valued keys. The clone stored after the first fill is missing label. On the next fill, duckSchema of the clone has fewer keys than the original → schema mismatch → fill blocked without any error. 
// BUG — deepClone in v17.0.1 (helpers/object.js):
export function deepClone(obj) {
  return JSON.parse(JSON.stringify(obj));
  // JSON.stringify({ id: 1, label: undefined }) → '{"id":1}'
  // 'label' key is silently dropped!
  // duckSchema(clone) !== duckSchema(original) → autofill blocked
}
How to observe the bug:
  1. Click "Fill row 0 → row 1" — row 1 gets a copy of row 0. ✓
  2. Click "Fill row 1 → row 2" — nothing happens. ✗
  3. The source data readout shows label is undefined in the original but missing in the clone stored after the first fill.
  4. Compare with the Fix page where both fills work. ✓
