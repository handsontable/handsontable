PR #12207 - Deep settings equality for non-plain object values in React wrapper

Build: Released v17.0.1 BUG PR #12207 Fix FIXED
BUG (v17.0.1): After clicking "Simulate re-render", updateSettings() is called with dataSchema and columns (new object references each render), resetting the autocomplete validator state. The htInvalid class disappears from the invalid cell.
How to reproduce:
  1. Click "Type 'INVALID' in Name cell" - the Name cell gets an invalid value.
  2. Wait ~200 ms - the htInvalid metric should turn YES (green).
  3. Click "Simulate re-render (sends dataSchema+columns again)" - this mimics what the v17.0.1 React wrapper does on every re-render triggered by afterChange.
  4. Observe: htInvalid turns NO (red) - the validator state was wiped by updateSettings.
Not affected in vanilla JS
// Vanilla JS: updateSettings() is called directly - no equality comparison
const hot = new Handsontable(container, {
  dataSchema: { name: '', createdAt: new Date() },
  columns: [{ data: 'name', type: 'autocomplete',
    source: ['Alice', 'Bob'], strict: true, allowInvalid: true }],
  licenseKey: 'non-commercial-and-evaluation',
});
// Direct call always works - no wrapper skipping logic
hot.updateSettings({ dataSchema: { name: '', createdAt: new Date() } });
Not affected in TypeScript (same as vanilla JS)

AFFECTED - BUG in v17.0.1
// BUG: v17.0.1 React wrapper uses reference equality for dataSchema/columns.
// Every re-render passes NEW object literals -> updateSettings() ALWAYS called
// with dataSchema+columns -> validator state RESETS -> htInvalid disappears.
import { HotTable } from '@handsontable/react-wrapper';
function App() {
  const [count, setCount] = React.useState(0);
  return (
    <HotTable
      {/* NEW object literal on every render -> reference !== prev -> ALWAYS sent to updateSettings */}
      dataSchema={{ name: '', createdAt: new Date() }}
      columns={[{ data: 'name', type: 'autocomplete',
        source: ['Alice', 'Bob'], strict: true, allowInvalid: true }]}
      afterChange={() => setCount(c => c + 1)}
      licenseKey="non-commercial-and-evaluation"
    />
  );
}
Vue 3 uses JSON.stringify comparison
<template>
  <HotTable
    :data-schema="schema"
    :columns="columns"
    @after-change="onAfterChange"
    license-key="non-commercial-and-evaluation"
  />
</template>
<script setup>
import { ref } from 'vue';
import { HotTable } from '@handsontable/vue3';
const schema = ref({ name: '', createdAt: new Date() });
const columns = ref([{ data: 'name', type: 'autocomplete',
  source: ['Alice', 'Bob'], strict: true, allowInvalid: true }]);
function onAfterChange() {
  // Vue uses JSON.stringify for comparison - Date serializes correctly.
  // Note: Set/Map objects serialize to {} in JSON (different Vue limitation).
  schema.value = { ...schema.value, createdAt: new Date() };
}
</script>
Angular uses direct property binding
// app.component.ts
import { Component } from '@angular/core';
@Component({
  selector: 'app-root',
  template: `<hot-table
    [dataSchema]="schema"
    [columns]="columns"
    (afterChange)="onAfterChange()"
    licenseKey="non-commercial-and-evaluation">
  </hot-table>`
})
export class AppComponent {
  schema = { name: '', createdAt: new Date() };
  columns = [{ data: 'name', type: 'autocomplete',
    source: ['Alice', 'Bob'], strict: true, allowInvalid: true }];
  onAfterChange() {
    // Angular change detection checks reference; updating schema triggers update
    this.schema = { ...this.schema, createdAt: new Date() };
  }
}
