DEV-1560 / #11365 — select editor drifts on scroll after close & reopen

Build: Released v17.0.1 BUG DEV-1560 Fix FIXED
BUG present (v17.0.1). When a type: 'select' cell editor is opened, closed, then reopened — scrolling the grid causes the editor to drift out of alignment with its cell.
  • 1st open → scroll: editor follows the cell correctly. ✓
  • Close (Esc), reopen → scroll: editor stays frozen at the old screen position. ✗
Root cause: SelectEditor.close() called clearHooks(), permanently stripping the afterScrollVertically / afterScrollHorizontally listeners. Since registerHooks() runs only once in init(), subsequent opens have no scroll tracking at all. 
// BUG — selectEditor.js close() in v17.0.1:
close() {
  this.selectWrapper.style.display = 'none';
  this.clearHooks();  // ← strips scroll listeners permanently on every close!
  super.close();
}
// registerHooks() is never called again → no repositioning after 1st close
Steps to reproduce:
  1. Double-click a cell in the "Status" column — a native <select> dropdown opens.
  2. Press Esc to close the editor without changing the value.
  3. Double-click the same cell again to reopen the editor.
  4. Click "Scroll down 10 rows" — the editor should follow the cell, but it won't. ✗
  5. The Editor position readout shows top is not updating. ✗
