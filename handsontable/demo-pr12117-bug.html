PR #12117 - Filters wrong column after manualColumnMove

Build: Released v17.0.1 BUG PR Fix FIXED
BUG present (released v17.0.1). After moving a column with manualColumnMove, the Filters plugin loses track of the filter when data changes. Root cause: #onAfterChange passes a visual index to hasConditions() which expects a physical index, so the condition lookup fails and the filter value list is never refreshed.

Reproduce the bug — follow these steps in order:

1Start: grid has columns Sold on (col 0), Product (col 1), Model (col 2).
2Move Sold on to last position (visual 0 → visual 2). Physical index stays 0.
3Apply a "by_value" filter on Sold on (now at visual col 2, physical col 0).
4Add a new row with a new date in Sold on.
5Check filter conditions — should still reference the correct (physical) column.
Press "Move column" to start the test scenario.
Event log:

You can also open the filter dropdown manually on Sold on column after each step to observe the bug visually.
